# chitty-1p-bridge — Design

- **Date**: 2026-05-02
- **Status**: v0.2 (concierge findings folded)
- **Author**: Claude (brainstorm session with @nb)
- **Spec URI**: chittycanon://docs/ops/specs/chitty-1p-bridge

## Context

ChittyConnect adopted the **Portal Pattern** for credentials: 1Password is cold/source-of-truth, Cloudflare Secrets Store is hot/runtime, and the Worker only ever reads env bindings (zero runtime 1P calls). The legacy 608-LOC `OnePasswordConnectClient` in chittyconnect is explicitly marked deprecated in `credential-broker.js`.

Two real gaps remain:

1. **1P → CF Secrets sync is deploy-time only.** Changes in 1P don't propagate until the next `wrangler deploy`. There is no continuous sync.
2. **VM-resident operator tooling has no canonical 1P client.** Each script on chittyserv-dev hand-rolls 1P access (or uses `op` CLI), creating drift and inconsistent error handling.

We considered integrating the official `@1password/connect` SDK directly into the chittyconnect Worker. Smoke-tested it — bundle works (+57 KiB gzipped under `nodejs_compat`) but the SDK's default HTTPClient uses Node `http`/`https` modules and `http.Agent`, which are fragile under `nodejs_compat`. More importantly, putting the SDK in the Worker would re-introduce the runtime 1P dependency the Portal Pattern was designed to eliminate.

This design instead places the SDK in a dedicated VM-resident service, **chitty-1p-bridge**, which serves both gaps without compromising the Portal Pattern.

## Scope

### In scope
- Node service running on chittyserv-dev as a systemd unit
- 1P → Cloudflare Secrets Store sync via cron poll + watchlist
- `chitty-op` CLI for operator-grade 1P access (get, list, otp)
- Reusable Node module other VM tools can import
- ChittyChronicle logging of every sync event and CLI invocation
- Phase 0 discovery of CF API token state (no operator credential ask)

### Out of scope (initial)
- Bi-directional sync (CF/KV → 1P) — deferred to Phase 3
- 1Password Events API integration — deferred to Phase 4
- Loopback HTTP API for non-Node tools — deferred until needed
- Replacing the legacy `OnePasswordConnectClient` in chittyconnect — out of scope; that stays as-is on its deprecation arc
- Synchronizing to Workers Secrets (legacy, not Secrets Store) — only Secrets Store is supported

### Components

#### 1. `@chittyops/1p-bridge` (Node module)
Thin wrapper around `@1password/connect`. Single instantiation point, ambient config from env. Exposes:
- `getItem(vault, item)` → full item
- `getField(vault, item, field)` → field value
- `getOtp(vault, item)` → TOTP code
- `listItems(vault)` / `listVaults()` → discovery
- `createItem(vault, builder)` / `updateItem(vault, item, mutator)` → write (Phase 3)

#### 2. `chitty-op` CLI
Entry points (commander or yargs):
- `chitty-op get <vault>/<item>/<field>` → prints field value
- `chitty-op list [vault]` → lists vaults or items in vault
- `chitty-op otp <vault>/<item>` → prints current TOTP
- `chitty-op sync status` → shows last sync results from chronicle
- `chitty-op sync run --dry-run` → executes one sync cycle without writing

#### 3. Sync daemon
Long-running systemd service. On each tick:
1. Load watchlist (`/etc/chitty-1p-bridge/watchlist.toml`)
2. For each entry, fetch from 1P, hash field value
3. Compare with cached hash in `/var/lib/chitty-1p-bridge/state.json`
4. If changed, push to CF Secrets Store via API
5. Update cached hash, log to ChittyChronicle
6. On error: log, alert, do not advance state (next tick retries)

#### 4. Watchlist (`/etc/chitty-1p-bridge/watchlist.toml`)

The bridge writes to the account-scoped Cloudflare Secrets Store. Binding those secrets into specific Workers is declared in each Worker wrangler.jsonc (the bridge does not modify Worker configs). The target_workers field in the watchlist is informational only — it tells operators which Workers consume this secret so the impact of a sync is visible in the chronicle log.

```toml
[[entry]]
vault = "infrastructure"
item = "cloudflare-api-key"
field = "credential"
target_account_id = "0bc21e3a5a9de1a4cc843be9c3e98121"
target_secret_name = "CLOUDFLARE_API_KEY"
target_workers = ["chittyconnect", "chittyrouter"]  # informational; see clarification note below

[[entry]]
vault = "integrations"
item = "mercury-aribia"
field = "api_token"
target_account_id = "0bc21e3a5a9de1a4cc843be9c3e98121"
target_secret_name = "MERCURY_TOKEN_ARIBIA_LLC"
target_workers = ["chittyconnect"]  # informational
```

Code-reviewed via PR. Adds-by-PR is the enforcement mechanism — no auto-discovery. Watchlist linter (\`npm run lint:watchlist\`) cross-checks entries against chittyconnect's \`wrangler.jsonc\` Secrets Store inventory and fails PRs that introduce drift.

**TOTP fields are rejected at watchlist load.** 1P normalizes TOTP differently on read vs write, so hash-on-value never stabilizes — the entry would re-sync every tick. The loader rejects any entry whose 1P field type is \`OTP\` with a clear error message. Use the \`chitty-op otp\` CLI for runtime TOTP retrieval instead.


## Data flow

### Sync flow (per tick)
```
cron tick
  └─▶ load watchlist
        └─▶ for each entry:
              ├─▶ SDK.getItem(vault, item)
              ├─▶ extract field STRING value (NEVER hash item envelope — updated_at/version/id mutate)
              ├─▶ compare with state cache
              ├─▶ if differs:
              │     ├─▶ CF API: PUT /accounts/:id/secrets_store/secrets/:name
              │     ├─▶ for each target_worker: ensure binding exists
              │     ├─▶ update state cache
              │     └─▶ chronicle.log({event: "secret.synced", entry, hash})
              └─▶ if error:
                    ├─▶ chronicle.log({event: "secret.sync_failed", entry, error})
                    └─▶ leave state cache untouched (retry next tick)
```

### CLI flow
```
chitty-op get infrastructure/cloudflare-api-key/credential
  └─▶ bridge module: getField("infrastructure", "cloudflare-api-key", "credential")
        └─▶ SDK.getItemByTitle(vault, item)
              └─▶ find field by label, return value
        └─▶ chronicle.log({event: "op.get", actor: nb, path})
```

## Phasing

### Phase 0 — Discovery (no code)
Verify the runtime prerequisites without asking the operator:
- Inventory existing CF API tokens for the account(s) in the watchlist; check for one with `Secrets Store:Edit` scope. Use the chittyconnect credential-broker / ChittyCommand to enumerate.
- Confirm 1P Connect server is reachable from the bridge's intended host (it already is — used by chittyconnect via the deprecated client).
- Confirm ChittyChronicle ingest endpoint is reachable.
- If a suitable CF token doesn't exist, mint one through the existing provisioning path (chittyconnect's enhanced-provisioner or manual via 1P → CF dashboard, depending on what discovery reveals). **No credential is requested from the operator.**

### Phase 1 — Foundation (low risk)
- New repo `chitty-1p-bridge` (already scaffolded)
- `package.json`, `@1password/connect` install
- Bridge Node module with read methods only
- `chitty-op get/list/otp` CLI
- `README.md` with operator quickstart
- Unit tests for the module

**Acceptance**: operator can run `chitty-op get <path>` from chittyserv-dev and get a credential.

### Phase 2 — Sync (the main feature)
- Watchlist file format + loader + validator
- Sync daemon entry point
- CF Secrets Store API client (minimal — `putSecret`, `listSecrets`)
- State cache (`/var/lib/chitty-1p-bridge/state.json`)
- ChittyChronicle logger
- systemd unit file (timer-based, every 5 min)
- `chitty-op sync run --dry-run` and `chitty-op sync status`

**Acceptance**: rotating an item in 1P propagates to the Worker's bound secret within 5 min; chronicle shows the event; `--dry-run` shows what would change without changing it.

### Phase 3 — Bi-directional (after Phase 2 has been boring for ≥30 days)
- Add SDK write methods to the Node module (`createItem`, `updateItem`)
- Hook `secret-rotation.js` post-rotation: when KV is updated, write the new value back to 1P
- Sync daemon detects the 1P update on its next tick. Loop prevention: the state-cache hash matches (value written back to 1P equals what 1P returns), so no re-push happens.

**Acceptance**: rotating GDrive token in chittyconnect updates 1P within one rotation cycle.

### Phase 4 — Events API (only if needed)
- Subscribe to 1P Events API (`onepassword-events.js` already scaffolds the client)
- On `item.updated` event for any watchlist item, trigger sync immediately
- Falls back to cron if event stream is down

**Acceptance**: change in 1P propagates to CF within 30 seconds.

## Operational concerns

### Auth model
- **1P Connect token**: stored on VM, scoped to the four canonical vaults (infrastructure/services/integrations/emergency). Read-only for Phases 1-2; read+write added at Phase 3.
- **CF API token**: stored on VM, scoped to `Secrets Store:Edit` for the specific account(s) in the watchlist. Discovered/minted in Phase 0.
- **No Worker tokens** — bridge never holds any service token a Worker uses.
- **Bridge token rotation policy**: 1P Connect token (read) and CF API token rotated quarterly. Phase 1-2: manual rotation by operator (run \`scripts/rotate-bridge-tokens.sh\` then restart unit). Phase 3+: bridge rotates its own CF token via the CF API, writes new token to 1P, picks it up on next tick. The 1P Connect token is rotated by an out-of-band operator action (cannot self-rotate the credential that authenticates to the credential store).

### Observability
- Every CLI invocation logged to ChittyChronicle with actor, path, result
- Every sync tick logged with summary (changed/unchanged/failed counts)
- Each individual change logged as its own event
- systemd journal available locally; chronicle is the durable record

### Failure modes
| Mode | Behavior |
|---|---|
| 1P Connect server down | Skip tick, log error, retry next tick |
| CF API down | Skip changed entries, leave state untouched, retry next tick |
| Watchlist parse error | Refuse to start; alert |
| State cache corrupt | Treat all entries as changed; resync once; log warning |
| Watchlist references missing 1P item | Log error per entry, skip, continue |
| Two bridges run simultaneously | State cache is local-only; safe but redundant. Add lockfile if we ever run more than one. |
| VM offline | No syncs occur during outage; operator runs \`chitty-op sync run\` after VM recovery. Documented as known SPOF. |
| Race with \`wrangler deploy\` | See § Deploy ordering below. |

### Deploy ordering (Phase 3+)

The CF Secrets Store API is not transactional per-secret; a bridge \`PUT\` racing a \`wrangler secret put --env production\` is last-writer-wins with no notification. Resolution rules:

- **Phase 1-2** (1P → CF only): If a deploy pushes a value to CF that does NOT yet exist in 1P, the bridge will overwrite it on the next tick with the stale 1P value. **Mitigation**: until Phase 3 lands, deploys MUST update 1P first, then run \`chitty-op sync run\` to push to CF, then deploy. Document this in chittyconnect deploy runbook.
- **Phase 3+** (bi-directional): Bridge treats any CF-side change with a hash unknown to its state cache as authoritative — it pulls the new value, writes it to 1P, then updates state. Race resolves via 1P winning the tie within ~5 minutes regardless of order.

### Repo conventions
- Lives at `github.com/CHITTYOS/chitty-1p-bridge`
- `CHARTER.md`, `CHITTY.md`, `CLAUDE.md` per ChittyOS compliance triad
- Registered with ChittyRegistry as a Tier 3 (Operational) service
- Deploys via systemd, not Workers — reflected in CHARTER

## Open decisions (tracked, not blocking)

1. **Watchlist file format** — TOML proposed; YAML or JSON would also work. Preference?
2. **State cache location** — `/var/lib/chitty-1p-bridge/state.json` (per-user) vs `/var/lib/chitty-1p-bridge/state.json` (system). Depends on whether daemon runs as the operator user or a service user.
3. **Lockfile/single-instance** — add now or defer until we actually run multiple instances?
4. **CLI distribution** — installed globally on VM via `npm link`, or invoked via `npx`?
